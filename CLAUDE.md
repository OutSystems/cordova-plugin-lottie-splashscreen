# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

This is a Cordova plugin that replaces the standard splash screen with animated Lottie splash screens for Android and iOS. The plugin bridges JavaScript API calls from the Cordova WebView to native implementations using Airbnb's Lottie libraries.

**For detailed architecture and design patterns**, see [ARCHITECTURE.md](./ARCHITECTURE.md) for the bidirectional event-driven bridge pattern, platform-specific configuration hierarchy, and external integration details.

**For development setup and contribution guidelines**, see [CONTRIBUTING.md](./CONTRIBUTING.md) for code standards, commit message format, and the pull request process.

## Repository Structure

```
src/
  android/        Kotlin implementation (LottieSplashScreen.kt, ColorHelper.kt, exceptions)
  ios/            Swift implementation (LottieSplashScreen.swift, UIColorHelper.swift)
www/              TypeScript source (lottie-splashscreen.ts)
dist/www/         Compiled JavaScript output (generated by tsc)
types/            TypeScript type definitions
example/          Working Cordova app demonstrating plugin usage
  config.xml      Example configuration with all available preferences
  www/            Demo HTML/JS
scripts/          Build automation scripts
plugin.xml        Cordova plugin configuration (platform blocks, dependencies)
```

The TypeScript source in `www/` compiles to `dist/www/` - the `dist/` directory is what gets published to npm. Each platform (Android/iOS) has completely independent native implementations with no shared code.

## Build and Test Commands

| Command | Purpose |
|---------|---------|
| `npm run build` | Compile TypeScript (`www/` → `dist/www/`) |
| `npm run lint` | Run all linters (Android + iOS + TypeScript) |
| `npm run lint:android` | ktlint on `src/android/*.kt` |
| `npm run lint:ios` | SwiftLint on `src/ios/*.swift` |
| `npm run lint:typescript` | ESLint on `www/*.ts` |
| `npm run release` | standard-version: bump version, update CHANGELOG, create git tag |

**Testing changes manually**: Use the `example/` project:
```bash
cd example
cordova platform add android  # or ios
cordova run android            # or ios
```

**Pre-commit automation**: Husky and lint-staged run Prettier on staged files automatically.

## Key Context for Agents

### Platform Requirements

**Android:**
- cordova-android >= 9.0.0 (Kotlin and AndroidX support required)
- For cordova-android < 10.0.0: set `GradlePluginKotlinEnabled` and `AndroidXEnabled` to `true` in `config.xml`

**iOS:**
- cordova-ios >= 5.0.0, iOS >= 11.0
- CocoaPods must be installed (`sudo gem install cocoapods && pod setup`)
- Valid `SwiftVersion` preference in `config.xml` (minimum 5.5)

See [FAQ.md](./FAQ.md) for common build errors and solutions.

### Configuration System

All plugin behavior is controlled via Cordova preferences in `config.xml` (see README.md for complete preference list). The plugin uses a three-tier preference fallback:
1. Dark mode suffix (`LottieAnimationLocationDark`)
2. Light mode suffix (`LottieAnimationLocationLight`)
3. Base preference (`LottieAnimationLocation`)

**Evidence**: See `getUIModeDependentPreference` in both `src/android/LottieSplashScreen.kt` and `src/ios/LottieSplashScreen.swift`.

Runtime overrides are possible via the JavaScript `show()` method, which accepts optional parameters that supersede `config.xml` values.

### Cross-Platform Timing Differences

**CRITICAL**: `LottieHideTimeout` and `LottieFadeOutDuration` have different units:
- **iOS**: Values in **SECONDS** (e.g., `0.5` is valid)
- **Android**: Values in **MILLISECONDS**

This is not a bug - it reflects the native platform APIs. Document this clearly when working with timeout-related preferences.

### Event-Driven Communication

Native code dispatches events back to JavaScript asynchronously:
- `lottieAnimationStart`, `lottieAnimationEnd`, `lottieAnimationCancel`, `lottieAnimationRepeat`
- Dispatched via `evaluateJavaScript` (iOS) or `evaluateJavascript` (Android)
- JavaScript listens via `document.addEventListener`

See the static initialization block in `www/lottie-splashscreen.ts` for listener setup.

### Concurrent Animation Guard

Only one animation can play at a time. Attempting to show a second animation throws:
- Android: `LottieSplashScreenAnimationAlreadyPlayingException`
- iOS: Error passed to callback

Check the `splashDialog` (Android) or `visible` flag (iOS) state before creating views.

### TypeScript Compilation

TypeScript must be compiled before publishing (`prepublishOnly` script). Compilation settings in `tsconfig.json`:
- Target: ES5
- Strict mode enabled (strict null checks, no implicit any, etc.)
- Output: `dist/www/` directory

The `dist/` directory is git-ignored but published to npm (see `.npmignore` vs `.gitignore`).

## Working with Native Code

### Android (Kotlin)

- Located in `src/android/`
- Lints with ktlint (4-space indentation, import ordering disabled)
- Uses Lottie Android 4.2.1 and AndroidX dependencies
- Key files: `LottieSplashScreen.kt` (main logic), `ColorHelper.kt` (hex parsing), `LottieSplashScreenExceptions.kt` (custom error types)

### iOS (Swift)

- Located in `src/ios/`
- Lints with SwiftLint (140 char line length)
- Uses lottie-ios ~> 4.4 via CocoaPods
- Key files: `LottieSplashScreen.swift` (main logic), `UIColorHelper.swift` (color parsing)

### Modifying plugin.xml

After editing `plugin.xml`, test installation in a fresh Cordova project to catch configuration errors before committing. The `version` attribute is auto-updated by the `release` script.

## Important Files Not to Modify

- `CHANGELOG.md` - Generated by standard-version
- `dist/` - Generated by TypeScript compiler
- `package-lock.json` - Only update via npm install/update

## Common Pitfalls

1. **Forgetting to build TypeScript**: Changes in `www/` won't take effect until you run `npm run build`
2. **CocoaPods not installed on iOS**: See FAQ.md for setup instructions
3. **AndroidX conflicts**: Requires cordova-android >= 9.0.0 or manual AndroidX flags
4. **Testing without example project**: Always test plugin changes in `example/` before publishing
5. **Line number references**: Never reference specific line numbers in documentation - they go stale immediately. Use function/class names instead.

## Useful Commands for Discovery

Find all preferences in example config:
```bash
grep -E '<preference name="Lottie' example/config.xml
```

List all exported TypeScript types:
```bash
grep -E 'export (type|interface|class)' www/lottie-splashscreen.ts
```

Check native dependencies:
```bash
# Android dependencies
grep '<framework src=' plugin.xml | grep android

# iOS dependencies
grep -A5 '<podspec>' plugin.xml
```
